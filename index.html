System Architecture
How the three ticket systems connect end-to-end
Every customer issue flows through three distinct systems. Understanding the boundaries — what lives where, and who updates what — is the single biggest source of visibility gaps today.
JSM — Customer Ticket (CDT)

Sub-tickets — CIPHER / CUR (Engineering)
⚡ Fast-track triggers (any stage): ESCALATION — founders involved; ESCALATED — formally escalated via CS; ROLLOUT_BLOCKERS — blocking a go-live; HYPERCARE — post-launch heightened monitoring.
↪ Stage 4 shortcut: if no engineering is needed, CDT closes the customer ticket directly without creating a CPS or sub-tickets.

0

Stage 1

Triaging

Ticket arrives in Open with Triaged = No · SLA: within 24 hours
CDT
1
Actions
  • 1
    Review the ticket. If it covers multiple distinct problems, split it into separate tickets before triaging, one ticket = one problem Not followed today
  • 2
    Determine Issue Category from the taxonomy below
  • 3
    Set Priority (P0/P1/P2/P3) based on customer sentiment, contract value, issue type, and whether the customer has explicitly flagged urgency
  • 4
    Set the Product Area field (customfield_10902 in Jira). This is a label-picker field, not a dropdown — type and select the correct value from the list below. Routing to the right POD depends on this field being set correctly.
  • 5
    Add an Internal Note with your initial RCA hypothesis, one or two sentences is enough
  • 6
    Set Triaged = Yes and route to the right next stage (see Hand Off below)

Auto-Triage Bot: Do Not Ignore
An automated triage bot classifies priority and team in ticket comments on ~54% of tickets. CDT must validate its output, if the bot's classification matches, use it. If it's wrong, correct it with an Internal Note explaining why. Do not silently discard it. The bot's signal is accurate more often than the manual priority field.

Status Name Note: Differs by Project Board
Different JSM project boards use different names for the same conceptual status. "Open State" (ANCS, MERCS, RCS, AHFCS, HECS) and "To Do" (EHCS, CPS, CIPHER) both mean the starting state. When this document says a status name, apply the equivalent in your project's board.
